# Module 06 · 双轴设计模式

## 这个模块解决什么问题

Claude Code 给了 Memory / Skills / Sub-Agents / Hooks / MCP 五大扩展点——但**怎么组合这些扩展点搭出一个 agent**没有现成答案。同样要做一个客服 agent，有人写成一条主链调几个 sub-agent，有人写成主从层级，有人写成多 agent 并行——表面上都「能跑」，但出问题时哪种结构最容易调、最经济、最不容易雪崩，差异巨大。双轴设计模式提供的就是这套**设计语言**：把 agent 系统在两个正交维度上分类（7 种能力 × 6 种拓扑 = 28 个模式格子），让你能对着自家系统说「我用的是 Hierarchical Retention + Tool Dispatch + Generator-Critic」，而不是「我用了一些 LangChain 和一些 prompt」。

## 学完你会

1. 看得懂双轴矩阵：纵轴 7 种能力（感知 / 记忆 / 推理 / 行动 / 反思 / 协作 / 治理）× 横轴 6 种拓扑（链式 / 路由 / 并行 / 编排 / 循环 / 层级）= 28 个具名模式
2. 拿到任何一个开源 agent 源码，能用**逆向五步法**（Detect → Classify → Filter → Map → Verify）拆解出它走了哪几个模式格子
3. 理解**复合错误公理**——95% 单步成功率到 10 步就只剩 60%——为什么这是 agent 工程的核心约束，以及 4 种工程响应（减少迭代 / 提单步成功率 / 加 verification / fail fast）

## Demo 怎么跑

**本模块是文档 + 图模块，没有可跑 demo**。

双轴框架的本质是**设计语言 + 选型方法**，运行时演示反而稀释概念。这个 module 教的是「看懂自家系统在矩阵上的位置」「读得懂别人 agent 的源码」「知道下一步该补哪个模式」，靠 PPT + 讲解 + 学员对自己系统的现场标注完成，不靠跑代码。

demo/ 目录下只放一份说明（[demo/README.md](demo/README.md)）说明这个设计意图，避免学员预期错位。

## 底层原理

### 双轴矩阵

```
                ┌──── 横轴 6 种拓扑 ────────────────────────┐
                │ 链式  路由  并行  编排  循环  层级       │
        ┌───────┼──────────────────────────────────────────┤
        │ 感知 │    P-1  P-2  P-3  P-4  P-5  P-6           │
        │ 记忆 │    M-1  M-2  M-3  M-4  M-5  M-6           │
   纵轴 │ 推理 │    R-1  R-2  R-3  R-4  R-5  R-6           │
   7 种 │ 行动 │    A-1  A-2  A-3  A-4  A-5  A-6           │
   能力 │ 反思 │    F-1  F-2  F-3  F-4  F-5  F-6           │
        │ 协作 │    C-1  C-2  C-3  C-4  C-5  C-6           │
        │ 治理 │    G-1  G-2  G-3  G-4  G-5  G-6           │
        └───────┴──────────────────────────────────────────┘
        42 格 - 14 空格 = 28 个有意义的模式
        14 个空格不是 bug，是 feature（某些能力×拓扑组合本身不成立）
```

每个格子对应一个有工业出处的具名模式（如 M-6 = Hierarchical Retention、A-2 = Tool Dispatch、F-1 = Generator-Critic）。完整 28 模式清单 + 工业出处见私库 `01-Agent设计模式之美/v4/final/01-范式觉醒/`，本 module 的 `pictures/` 里有渲染好的矩阵 PNG。

### 复合错误公理（为什么 agent 这么难）

```
单步成功率    1 步   3 步    5 步    10 步
   95%       95%    86%     77%     60%
   99%       99%    97%     95%     90%
```

这两行数字解释了为什么 agent 容易雪崩：单步看起来很高的 95%，到 10 步就只剩 60%。所以 agent 工程的核心约束是「步数 × 单步成功率」。4 种工程响应：

1. **减少迭代步数** —— 用更直接的拓扑（链式 / 路由）代替循环
2. **提高单步成功率** —— 更强模型 / 更好 prompt / 更少模糊空间
3. **加 verification 关卡** —— 关键节点用 Generator-Critic 双跑校验
4. **Fail fast** —— 失败立刻停，不要让错误在后续步骤里被放大

### 逆向五步法（拿到陌生 agent 怎么读源码）

1. **Detect** —— 找到主循环（while / for / recursive call）在哪
2. **Classify** —— 这个循环属于 6 种拓扑里的哪种
3. **Filter** —— 砍掉 70% 的 boilerplate（日志、错误处理、配置），看骨架
4. **Map** —— 把每个组件落到矩阵的某个格子
5. **Verify** —— 回到源码验证你的判断（最关键，前 4 步可能猜错）

## 在 Claude Code 中怎么落地

双轴框架是设计语言，不是 Claude Code 本身的扩展点。它的作用是：
- 告诉你 M01-M05 学的扩展点该**怎么组合**
- 告诉你拿到别家 agent（CrewAI / LangGraph / AutoGen / Claude Code 自身）能读懂源码
- 告诉你自家系统下一步该补哪个模式（盲区识别）

**典型应用**：用双轴矩阵看 Claude Code 自身——它走了哪些格子？答案见 PPT 后段三大系统对比表（Claude Code / Manus / OpenClaw × 6 模式）。
